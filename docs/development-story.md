# UML Canvas 開発ストーリーと技術解説

この文書は、UML Canvasを作った背景、採用した技術、実装上の判断、現在できることと
できないことを記録したものです。GitHub上の開発資料であると同時に、後日Qiitaなどで
開発記事を書く際の一次資料として利用できるよう、経緯と判断理由を含めて整理しています。

## 1. 何を解決したかったのか

GPTやCodexに業務フローをPlantUMLで生成してもらうと、数百行に及ぶ巨大なコードに
なることがあります。生成された画像をそのまま見るだけでは、次の問題がありました。

- 全体を表示すると文字が小さく、細部を確認できない
- 拡大すると図のどこを見ているのか分からなくなる
- 縦長のアクティビティ図では、現在の処理がどの主体に属するか見失う
- PlantUMLのエラーが画像として表示され、文章をコピーしてAIへ返せない場合がある
- AIが作った図の軽微な修正だけでも、巨大なソースコードから該当箇所を探す必要がある
- 一般的なオンラインレンダラーへ、社内フローや未公開情報を送信したくない

そこで、PlantUMLコードを貼り付けるだけで、大きな図をキャンバスのように拡大・縮小・
移動でき、さらに図を直接触って微修正できるブラウザツールを作ることにしました。

## 2. 名前について

当初は機能をそのまま表す「PlantUML Viewer」という仮称でした。しかし、同名・類似名の
ツールが多く、単なる閲覧ソフトよりも「大きな図を広げて触る場所」という性格が強く
なったため、名称を「UML Canvas」に変更しました。

PlantUMLは入力形式と描画エンジンを示す言葉として説明文や検索タイトルに残し、製品名は
短い「UML Canvas」にしています。

## 3. 現在できること

### 3.1 貼り付けと描画

- PlantUMLコードを貼り付けて描画
- MarkdownのPlantUMLコードフェンスを貼り付けた場合は囲みを自動除去
- 描画エラーを選択・コピー可能なテキストとして表示
- PNGとSVGで保存
- 入力コードをブラウザのローカルストレージへ保存

### 3.2 大きな図の閲覧

- マウスホイールによる拡大・縮小
- ドラッグによるキャンバス移動
- 全体表示、100%表示、全画面表示
- 操作バー自体をドラッグして邪魔にならない位置へ移動
- アクティビティ図のスイムレーン名を追従表示

### 3.3 描画上限の扱い

ブラウザで極端に大きな図を無制限に描画すると、メモリ不足やタブのフリーズにつながります。
そのため初期上限は縦横8,192px、利用者が指定できる最大値は32,768pxとしています。

通常時には上限設定を見せず、実際に上限超過エラーが発生したときだけ「上限を変更して
再描画する」画面を表示します。必要な寸法を示したうえで任意のpx値を入力できるため、
安全性と巨大図への対応を両立しています。

### 3.4 箱の編集

標準的なアクティビティ記法の `:処理;` を解析し、図上の箱とPlantUMLソースを対応付けます。
編集モードでは次の操作ができます。

- 箱の文章変更
- 箱の追加、複製、削除
- 前後への移動
- 主体（スイムレーン）の変更
- 上下ドラッグによる順番変更
- 左右ドラッグによる主体変更
- 箱同士の間へ新しい箱をドロップ
- ドロップ前の挿入位置・移動先レーン表示
- 編集前後のコード差分確認
- 元に戻す・やり直す

### 3.5 IF・while・forkの編集

箱だけでなく、標準的な制御構造も安全な範囲で編集できます。

- 選択した箱の後へIF条件分岐を追加
- 図上のIFひし形を直接クリック
- IF条件と「はい／いいえ」ラベルを変更
- 箱を既存IFの「はい／いいえ」側へ移動
- IFの枠だけを解除し、両側の箱を直列に残す
- IF内の箱を含めて分岐全体を削除
- 図上のwhile条件をクリックし、条件・継続・終了ラベルを変更
- forkの枝を追加、削除、並べ替え
- 同じ直列フロー内で次の接続先を変更

IF追加後は編集画面を閉じず、そのまま条件変更や削除ができます。削除も履歴に記録されるため、
誤操作した場合は「元に戻す」で復元できます。

## 4. 描画の仕組み

### 4.1 サーバーへPlantUMLコードを送らない

UML Canvasは、PlantUMLコードを外部のPlantUML描画サーバーへ送信しません。
`@plantuml/core`が提供するTeaVM版PlantUMLとGraphvizのブラウザ実行資産を読み込み、利用者の
ブラウザ内でSVGを生成します。

本番のCloudflare PagesはHTML、CSS、JavaScript、描画エンジンなどの静的ファイルを配信する
役割です。利用者が貼り付けたPlantUMLコードをCloudflareやOpenAIのサーバーで描画する構成
ではありません。

### 4.2 SVGの安全な表示

生成したSVGは不要・危険な要素や属性を除去してからBlob URLへ変換し、`img`として表示します。
SVGをページDOMへ直接埋め込まず画像として隔離することで、入力由来のSVGがアプリ本体へ
干渉しにくい構成にしています。

### 4.3 図を直接編集できる理由

PlantUMLのSVG自体を自由に書き換えても、元のPlantUMLコードへ正しく戻すことはできません。
そこでUML Canvasでは、PlantUMLコードを常に正本とし、次の流れで編集します。

1. ソースコードから箱、スイムレーン、IF、while、forkの構造を解析する
2. 描画後のSVGから文字と図形の位置を読み取る
3. ソース側のラベルとSVG側の文字を照合する
4. 箱やひし形の位置へ、透明な操作ボタンを重ねる
5. 利用者の操作をPlantUMLコードの変更へ変換する
6. 変更後のコードから図を再描画する

画面上では図形を直接触っているように見えますが、実際には毎回PlantUMLソースを安全な形で
書き換えています。この方式により、編集後のコードをそのまま保存したり、AIへ返したりできます。

## 5. なぜ完全な自由描画にしなかったのか

PlantUMLにはIF、while、fork、スイムレーン、noteなど多くの構造があり、見た目上は近くても
ソース上では別の分岐・ループに属している場合があります。自由に線をつなぎ替えたり、箱を
どこへでも移動できるようにすると、次の問題が起こります。

- `endif`や`endwhile`との対応が壊れる
- 箱が意図しない分岐へ入る
- スイムレーンの主体が変わる
- 見た目は描けても、再利用できるPlantUMLコードへ戻せない

そのため、現在は「確実にPlantUMLへ戻せる操作」を優先しています。分岐・ループの境界を
越える危険な移動は拒否し、接続先変更も同じ直列フロー内の順番変更として実現しています。

この制約は機能不足というより、AIが生成した巨大コードを壊さずに人間が微修正するという
用途に合わせた設計判断です。

## 6. 開発を段階化した理由

編集機能は、壊れにくい順に三段階で追加しました。

### 第1段階：箱の微修正

箱の選択、文章変更、追加、削除、複製、前後移動、主体変更、Undo／Redo、コード差分を実装。
まずはクリック中心の操作で、ソース解析と再描画の基礎を固めました。

### 第2段階：ドラッグ編集

上下ドラッグによる順番変更、左右ドラッグによる主体変更、箱同士の間への追加、ドロップ位置の
プレビューを実装。誤操作を減らすため、ドラッグを始めた方向から操作意図を判定します。

### 第3段階：構造編集

IF追加、はい／いいえ側への移動、接続先変更、while・fork専用編集を追加。その後、IFそのものを
図上から選べない問題を受け、ひし形のクリック、条件変更、分岐解除、全体削除も追加しました。

段階ごとにブラウザ上で実際にコードを変更し、再描画後のソースと操作パネルを確認してから
公開しています。

## 7. 実際の利用から生まれた改善

このツールの主要機能は、最初からすべて決めて作ったものではありません。実際の巨大な
PlantUMLを貼り付け、困った点を一つずつ改善してきました。

- Markdownのコードフェンスごと貼り付けると、PlantUMLがコードフェンスの先頭行を未知の
  命令として解釈したため、描画前にコードフェンスを取り除く処理を追加
- エラーがSVG画像内に描かれてコピーできなかったため、エラーを通常のテキストとして表示
- 4,096pxのブラウザ描画上限へ到達したため、実行資産の上限処理を調整し、エラー時だけ
  任意pxで再実行できる画面を追加
- 固定された操作バーが図を隠したため、操作バーそのものをドラッグ可能に変更
- 縦長の図で主体名を見失ったため、現在位置のスイムレーン名を追従表示
- 編集確定のたびにパネルが閉じると連続修正しづらいため、編集画面と選択状態を維持
- IFを追加できてもひし形を選択・削除できなかったため、図上クリック、条件編集、分岐解除、
  全体削除を追加

要件表だけで完成を決めず、実物の図で「次に面倒な一手」を見つけて減らしていったことが、
現在の操作性につながっています。

## 8. 技術構成

- React 19
- Next.js互換App Router
- Vinext / Vite
- TypeScript
- `@plantuml/core`（TeaVM版PlantUML + Graphviz）
- Lucide React
- Cloudflare Pages
- GitHub Actions
- Wrangler

アプリケーションは静的サイトとしてビルドでき、Cloudflare Pagesへ配置されます。専用APIや
データベースは使用していません。

## 9. 公開・更新の流れ

本番ソースの正本はGitHubの `USS-Meigetsu/plantuml-viewer` リポジトリにある `main` ブランチです。

1. GPT Sites上で変更を依頼
2. ソースを修正
3. ESLintとCloudflare Pages向け静的ビルド・テストを実行
4. GitHubの `main` へ直接反映
5. GitHub Actionsがビルドとテストを再実行
6. Wranglerが既存のCloudflare Pagesプロジェクトへデプロイ
7. `https://plantuml.massivedyno.com/` で公開結果を確認

Pull Requestを必須にせず、小さな変更を短い経路で公開できるようにしています。一方で、過去の
OpenAI Sites版とCloudflare移行時点のブランチを残し、切り戻せる状態を維持しています。

## 10. ローカルでの確認方法

必要環境はNode.js 22.13以上とnpmです。

```bash
git clone https://github.com/USS-Meigetsu/plantuml-viewer.git
cd plantuml-viewer
npm install
npm run dev
```

主な検証コマンドは次のとおりです。

```bash
npm run lint
npm run test:pages
```

`npm run test:pages`はPlantUML実行資産を準備し、静的サイトをビルドしたうえで、Cloudflare Pages
向け成果物の基本テストを実行します。

## 11. 現在の制約

- 主な編集対象は標準的なアクティビティ図
- 特殊なPlantUML記法や複雑な入れ子は、表示できても視覚編集の対象外になる場合がある
- 同じ文章の箱が多数ある図では、SVGとの位置照合が難しくなる場合がある
- 自由な線描画や任意ノード間の接続はできない
- forkやwhileの複雑な入れ子は安全性を優先して編集範囲を限定
- 入力内容は端末間で同期されず、同じブラウザのローカルストレージにのみ保存される
- 非常に大きな図は、上限を上げるほどブラウザのメモリを多く消費する

## 12. 今後の候補

- IF・while・forkの入れ子構造をより分かりやすく表示するナビゲーション
- 同一ラベルが多い図での図形照合精度向上
- 編集対象外の構文と理由を画面上で説明
- 編集操作から生成されるコード差分の説明強化
- PlantUMLコードまたは編集履歴のファイル保存・読込
- キーボード操作とアクセシビリティの強化
- 巨大図の部分表示や仮想化によるメモリ使用量削減

## 13. Qiita記事へ展開するときの素材

記事では、次の順番にすると開発意図が伝わりやすくなります。

1. AIが巨大なPlantUMLを生成してくれるようになった一方、確認と微修正がつらかった
2. 画像出力ではなく、拡大・移動できるキャンバスを作った
3. 機密性を考え、サーバーレンダリングではなくブラウザ内描画を選んだ
4. エラーコピー、描画上限、移動式ツールバー、スイムレーン追従を利用中の不満から追加した
5. 表示ツールから、PlantUMLコードを壊さない視覚編集ツールへ発展した
6. SVGを直接編集せず、ソースを正本として再描画する設計にした
7. 自由度より安全な微修正を優先した
8. GitHub ActionsとCloudflare Pagesで更新を自動公開した

### 記事タイトル案

- AIが生成した巨大PlantUMLを、ブラウザで確認・微修正できる「UML Canvas」を作った
- PlantUMLを画像で眺めるのがつらいので、ズーム・移動・図上編集できるツールを作った
- GPTが生成したPlantUMLを壊さず直すため、SVGではなくソースを書き換えるUIを作った

### 用意すると分かりやすい画像

- 巨大なスイムレーン図を縮小表示した画面
- 拡大して移動している画面
- スイムレーン名が追従している画面
- 箱をドラッグしている途中の挿入位置表示
- IFのひし形をクリックして編集している画面
- 編集前後のPlantUMLコード差分
- 描画上限エラーから任意pxで再実行する画面
